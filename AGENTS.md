# Agent Development Notes

## Source of Truth

- Edit TypeScript source under `src/**/*.ts`.
- Do not edit `main.js` directly. It is generated by `npm run build`.
- Commit generated `main.js` after source changes because Obsidian loads it directly.

## Expected Workflow

When changing plugin behavior:

1. Read the relevant source in `src/`, not the generated `main.js`.
2. Add or update tests in `test/` for domain behavior.
3. Run `npm test`.
4. Run `npm run build`.
5. Verify root release artifacts still exist: `manifest.json`, `main.js`, `styles.css`.

## Dependencies

- Use project-local npm dependencies.
- Run `npm ci` after cloning or when `node_modules/` is missing.
- Do not install TypeScript, Vitest, esbuild, or Obsidian typings globally.
- Treat `node_modules/` as the local dependency environment, similar to Python `.venv/`.
- Use package scripts such as `npm test` and `npm run build` so commands resolve local tools from `node_modules/.bin/`.

## Project Boundaries

- Keep Obsidian runtime compatibility: the release artifact is CommonJS `main.js` at the plugin root.
- Avoid introducing a UI framework. The current view/modal code uses Obsidian DOM APIs.
- Prefer extracting pure logic into `src/domain/` before touching UI code.
- Keep settings data compatible with existing `data.json`; do not rename persisted keys unless a migration is added.
- Do not change release workflow assumptions without checking that releases still attach `manifest.json`, `main.js`, and `styles.css`.

## Testing Expectations

- Domain changes need Vitest coverage.
- Parser, date, servings, nutrition, shopping, and meal-plan mutation logic should be tested in `test/domain/`.
- UI-only changes should still run `npm test` and `npm run build`; add domain tests if the UI change alters business logic.
